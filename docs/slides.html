04 Architecture

系統架構

IoT Sensor（mock_sensors.py） └─ APScheduler 每 5 秒輪詢 ├─ 正常 → 繼續輪詢 └─ 異常 → pending_proactive → st.rerun() ↓ CareAgent（gpt-4o-mini）— 主控 Orchestrator ├─ 工具：get_vitals / get_sleep_report / get_medication_schedule ├─ 趨勢問題 → AnalysisAgent（Gemini 2.5 Flash）→ get_health_trend(N) └─ 數值異常 → AlertAgent（gpt-4o-mini）→ send_emergency_alert(reason) └─ schedule_followup(reason) → Timer → N 分後重入

警報閾值：心率 >120 或 <50 bpm ／ 血氧 <90% ／ 收縮壓 >140 mmHg

05 Multi-Agent Design

三個 Agent 的分工

🤖 CareAgent — 主控 Orchestrator
gpt-4o-mini · Session 記憶（_AgentWithMemory）
主對話、委派決策、Session 記憶；每則訊息只說一件事，用名字不用「用戶」
get_vitals get_sleep_report get_medication_schedule
🧪 AnalysisAgent — 深度趨勢分析
Gemini 2.5 Flash · Stateless（無記憶，每次獨立）
近 N 天生理趨勢分析、異常模式識別、結構化摘要回傳 CareAgent
get_health_trend(days=N)
🚨 AlertAgent — 緊急評估與通報
gpt-4o-mini · Stateless · cron-as-tool 自排程
確認最新數值、評估是否通報、執行通報、自排程 N 分後追蹤確認
get_vitals send_emergency_alert schedule_followup ✦

✦ 最小權限原則：每個 Agent 只持有它需要的工具，send_emergency_alert 不授予 CareAgent

06 Cron-as-Tool

Agent 自排程追蹤機制

AlertAgent 通報後，不依賴外部 Cron——Agent 自己呼叫工具排程追蹤

AlertAgent 通報完成 └─ 呼叫 schedule_followup(reason) └─ threading.Timer(N×60s, _enqueue) └─ followup_queue.append(msg) APScheduler 下一個 5s tick └─ pop_followup() 取出訊息 └─ pending_proactive = msg └─ CareAgent 主動追蹤確認
靈感來源
OpenClaw 的 cron-as-first-class-tool 概念：把「排程」當成 Agent 可呼叫的工具，而非外部基礎設施

Sidebar 設定

追蹤間隔 1–120 分鐘可即時調整，不需重啟 Agent；tool 呼叫時才讀全域設定值

追蹤訊息範例

「阿嬤，剛才心跳有點快，現在感覺怎麼樣？有好一點嗎？」
