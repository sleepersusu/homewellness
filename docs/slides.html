09Cron-as-Tool

Agent 自排程追蹤機制

AlertAgent 通報後，自己呼叫工具排程未來追蹤——不依賴外部 Cron

AlertAgent 通報完成 └─ 呼叫 schedule_followup(reason) └─ from agent.scheduler_tools import ... └─ actual = delay or get_default_followup_minutes() └─ threading.Timer(actual×60, _enqueue) └─ followup_queue.append(msg) APScheduler 下一個 5s tick └─ pop_followup() 取出訊息 └─ pending_proactive = msg └─ CareAgent 主動追蹤確認
靈感：OpenClaw cron-as-first-class-tool
把「排程」當成 Agent 可呼叫的工具，而非外部基礎設施

Sidebar 即時設定

追蹤間隔 1–120 分鐘可即時調整，不需重啟 Agent。tool 呼叫時讀全域設定 get_default_followup_minutes()

scheduler_tools.py

_followup_queue: list[str] 模組級佇列，threading.Timer 背景執行緒寫入，APScheduler 主執行緒 pop_followup() 讀取

生產對應

threading.Timer → AWS EventBridge Scheduler（持久化，服務重啟不丟失）

10Session Memory

Session 記憶設計

不使用 RunnableWithMessageHistory（LangChain 1.x 已 deprecated）
改用自建 _AgentWithMemory wrapper（agent/memory.py）
_AgentWithMemory.invoke(input, session_id) ├─ _store[session_id] → InMemoryChatMessageHistory ├─ history.messages + HumanMessage(input) ├─ graph.invoke({messages: [...]}) ├─ AIMessage → history.add_messages() └─ 聚合 usage_metadata（Token 統計） └─ return {output, usage_metadata}

記憶什麼

對話中說過的症狀 · 確認服藥 · 情緒狀態 · 家人提到的事 → 不重複詢問，對話有連貫感

子代理不加記憶

AnalysisAgent / AlertAgent 不包 memory wrapper，每次 stateless 獨立呼叫。只有 CareAgent 有 Session 記憶。

Token 聚合邏輯

從 LangGraph result 的每個 AIMessage.usage_metadata 彙整 input/output tokens，相容 OpenAI 與 Gemini 兩種格式

清空時機

Sidebar 切換模型組合時自動清空對話記錄，st.session_state.messages = [] + _store.clear()

11Prompt Engineering

三個 System Prompt 的設計

build_care_prompt()
CareAgent 主對話

— 姓名、年齡、病史動態注入
— 用藥計畫（藥名 + 時間 + 是否隨餐）
— 對話規則 8 條（語氣、禁止診斷、委派條件）
— 何時 call_analysis_agent / call_alert_agent

build_analysis_prompt()
AnalysisAgent 趨勢分析

— 警報閾值參考動態注入
— 分析規則：使用 get_health_trend 後分析
— 輸出格式：趨勢方向 + 異常點 + 建議
— 回傳簡潔（3–5 行），給 CareAgent 用

build_alert_prompt()
AlertAgent 緊急通報

— 緊急聯絡人姓名 + 電話動態注入
— 觸發條件（閾值）動態注入
— 流程：get_vitals → 評估 → send_emergency_alert → schedule_followup
— 回應簡短有力，不超過 3 行

設計原則：所有病患資料從 health_profile.json 動態讀取，不硬寫在程式碼中。更換病患只需換一個 JSON 檔案。

13Proactive Mechanism

Pending Proactive 機制

Streamlit 是同步渲染——背景執行緒無法直接更新 UI。用 session_state 作為訊息佇列橋接非同步事件。

# 背景執行緒（APScheduler） def _proactive_monitor(): followup = pop_followup() # 先查 cron 佇列 if followup: session_state["pending_proactive"] = followup return vitals = get_mock_vitals() if 超閾值: session_state["pending_proactive"] = "（系統觸發）偵測到..." # 主執行緒（每次 rerun 頂部） if session_state.pending_proactive: _msg = session_state.pending_proactive session_state.pending_proactive = None # 防重複 _resp, _stats = invoke_agent(..., _msg) st.rerun()
設計巧思：scheduler_started flag 防止 Streamlit rerun 重複建立 APScheduler

優先順序

cron followup 佇列優先於 IoT 閾值觸發：先 pop_followup()，有則直接 return，確保追蹤訊息不被異常偵測覆蓋

系統觸發標記

所有 proactive 訊息加 「（系統觸發）」 前綴，讓 Agent 區分「背景觸發」與「長者輸入」，調整回應語氣

防重複觸發

寫入 pending_proactive 後即清空（設為 None），避免下一個 rerun 重複呼叫 Agent

17Lazy Import · 架構防護

循環 Import 防護機制

問題：health_agent → analysis_agent → tools → health_agent 循環依賴
# ❌ 模組頂端 import（循環 import 報錯） from agent.analysis_agent import build_analysis_agent from agent.alert_agent import build_alert_agent # ✅ 函式體內 lazy import（正確） def build_agent(care_model, analysis_model, alert_model): from agent.analysis_agent import ( build_analysis_agent, invoke_analysis_agent) from agent.alert_agent import ( build_alert_agent, invoke_alert_agent) analysis_graph = build_analysis_agent() alert_graph = build_alert_agent() # 子代理包成 @tool（閉包捕捉 graph） @tool def call_analysis_agent(query: str) → str: return invoke_analysis_agent(analysis_graph, query) @tool def call_alert_agent(situation: str) → str: return invoke_alert_agent(alert_graph, situation)
原理：Python 只在函式被呼叫時才執行函式體內的 import，此時模組已完整載入，不會觸發循環依賴

閉包捕捉（Closure）

子代理 @tool 函式定義在 build_agent() 內，透過閉包捕捉 analysis_graph / alert_graph 實例，每次呼叫都用同一個 graph 物件

子代理 stateless 設計

AnalysisAgent / AlertAgent 無記憶 wrapper，每次 graph.invoke() 都是全新 stateless 呼叫。只有 CareAgent 有 session 記憶。

schedule_followup lazy import

tools.py 的 schedule_followup 函式體內才 import scheduler_tools，同樣避免循環依賴問題

24Design Principles

設計原則總覽

01
Proactive over Reactive
IoT 異常偵測主動觸發，把「等待求救」的責任從長者身上移走
02
Vibe First（語氣先行）
每則訊息只說一件事，用名字、鼓勵語氣，不作醫療診斷——語氣設計寫進 system prompt，不是事後微調
03
最小權限工具分配
send_emergency_alert 只給 AlertAgent；get_health_trend 只給 AnalysisAgent。降低誤觸風險，職責清晰
04
Cron-as-Tool（Agent 掌控時間軸）
AlertAgent 呼叫 schedule_followup 自排程追蹤，Agent 的行動不止於當下對話
05
成本可見性從第一天起
Token 追蹤與效能監控是 PM 對工程成本的基本責任，不是優化期才做的事
06
不用 Deprecated API，技術選型考慮維護成本
RunnableWithMessageHistory 已廢棄，自建 _AgentWithMemory；lazy import 解循環依賴
